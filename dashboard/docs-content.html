GitHub Reference

How Back Office is organized across repositories.

The GitHub reference explains the boundary between the Back Office control-plane repo and the product repos it audits, scaffolds, and reports on.

Repository split

Back Office repo owns the audit framework, dashboard UI, aggregators, workflow scaffolding templates, sync scripts, runner configuration conventions, and documentation pages.

Target product repos own their application code, local lint/test/build commands, and the generated GitHub Actions files Back Office scaffolds into place.

app source

deploy code
product docs

What should change where

  • Change config/targets.yaml in Back Office when you need to register a repo, update audit context, or define lint, test, and deploy commands.
  • Change the local active runner with python3 scripts/backoffice-cli.py activate-runner --runner ..., which writes config/agent-runner.env for your operator machine.
  • Change templates/github-actions/ in Back Office when you want to improve the generated CI, preview, CD, or nightly workflow templates.
  • Change product repo source code when Back Office findings identify issues or product work that belongs to that repo.
  • Review generated workflow files inside the target repo before committing them. Back Office scaffolds them; it does not auto-merge them into production repos.

Scaffolded workflow files

Back Office treats GitHub Actions as a generated starting point, not a hidden control plane. The output is meant to be reviewed, edited, and committed in the product repo.

python3 scripts/backoffice-cli.py scaffold-workflows --target bible-app
  • Generated files land in the target repo under .github/workflows/.
  • The scaffold uses target-level commands from config/targets.yaml.
  • If a repo has special deploy constraints, edit the generated workflow in that repo instead of overfitting the shared template.

CI/CD Reference

Delivery should be automated, reviewable, and hard to misuse.

This is the Back Office release standard for audited repos and for Back Office itself: validate every change, keep preview and production distinct, and protect production with explicit gates.

Required workflow set

  • CI: run on pull requests and non-main pushes with shell checks, Python validation, lint, tests, and optional coverage when configured for the target.
  • Preview: produce a reviewable dashboard artifact for pull requests.
  • CD: validate first, then deploy production only from main through GitHub OIDC + AWS.
  • Nightly: rerun CI and refresh delivery metadata artifacts on a schedule.

Release guardrails

  • Use workflow concurrency so stale CI and preview runs get canceled instead of consuming noise and time.
  • Add job-level timeouts to deploy and nightly workflows so stuck jobs fail fast.
  • Run smoke tests immediately after production deploys.
  • Use protected GitHub environments with required reviewers for production.
  • Prefer an overnight PR or queued release candidate over unattended production promotion when confidence is low.

Reference flow

Back Office audit
  -> prioritized work
  -> branch or PR
  -> CI gate
  -> preview validation
  -> review
  -> protected production deploy
  -> smoke test
  -> dashboard refresh

The core rule is simple: automate analysis and verification aggressively, automate production promotion carefully.

Back Office deployment paths

  • Repo deploy path: push to main in the Back Office repo and let GitHub Actions handle the dashboard deploy workflow.
  • Direct sync path: run bash scripts/sync-dashboard.sh when you intentionally want to publish the static dashboards and payloads from a local operator environment.
  • The direct sync path should still respect the same discipline: tests first, aggregation second, upload third.

CLI Reference

The portable operating surface for Back Office.

The CLI is the stable entry point for operators and coding agents. It wraps the underlying workflow, scaffold, test, deploy, and sync scripts into a single command surface.

Primary commands

python3 scripts/backoffice-cli.py setup --write-missing-configs
python3 scripts/backoffice-cli.py runners
python3 scripts/backoffice-cli.py activate-runner --runner codex --mode stdin-text
python3 scripts/backoffice-cli.py list-targets
python3 scripts/backoffice-cli.py refresh
python3 scripts/backoffice-cli.py deploy
python3 scripts/backoffice-cli.py test
python3 scripts/backoffice-cli.py audit --target bible-app --departments qa,product
python3 scripts/backoffice-cli.py audit-all
python3 scripts/backoffice-cli.py scaffold-workflows --target bible-app
python3 scripts/backoffice-cli.py quick-sync --department product --repo bible-app

Command intent

  • setup checks available agent CLIs, lists agent scripts and prompts, creates missing config files if asked, and shows what to change next.
  • runners shows which runner CLIs are installed locally and what runner Back Office is currently set to use.
  • activate-runner writes the active runner choice to config/agent-runner.env so future audits use it automatically.
  • list-targets shows the configured audit targets from config/targets.yaml.
  • refresh rebuilds dashboard payloads from existing results without rerunning agents.
  • deploy publishes dashboards and payloads with scripts/sync-dashboard.sh.
  • test runs the Back Office test gate.
  • audit runs one configured target through selected departments.
  • audit-all runs multiple configured targets through the selected departments.
  • scaffold-workflows writes starter GitHub Actions workflows into a configured target repo.
  • quick-sync pushes one department's dashboard payload for one repo-scoped target.

Recommended operating flow

python3 scripts/backoffice-cli.py setup --write-missing-configs
python3 scripts/backoffice-cli.py list-targets
python3 scripts/backoffice-cli.py audit --target back-office --departments qa,product
python3 scripts/backoffice-cli.py audit-all
python3 scripts/backoffice-cli.py refresh
python3 scripts/backoffice-cli.py test

Use this pattern when you want the CLI to behave like a documented operator surface: confirm the target set, run one deliberate audit lane, rebuild the dashboards, then confirm the repo still passes its test gate.

When to use the CLI instead of Make

  • Use the CLI when you want one documented entry point for humans and agents.
  • Use the CLI in docs and runbooks when you want command examples to stay stable even if lower-level scripts shift.
  • Use audit-all with no flags to run every configured target: python3 scripts/backoffice-cli.py audit-all.
  • Use Make when you specifically want short aliases inside the Back Office repo.
  • Use lower-level scripts only when you are developing those scripts directly or debugging their internals.
  • Use metrics.html when you want code-quality and agent-usage reporting instead of repo-by-repo findings pages.
