How Back Office is organized across repositories.

The GitHub reference explains the boundary between the Back Office control-plane repo and the product repos it audits, scaffolds, and reports on.

What should change where

  • Change config/targets.yaml in Back Office when you need to register a repo, update audit context, or define lint, test, and deploy commands.
  • Change the local active runner with python3 scripts/backoffice-cli.py activate-runner --runner ..., which writes config/agent-runner.env for your operator machine.
  • Change templates/github-actions/ in Back Office when you want to improve the generated CI, preview, CD, or nightly workflow templates.
  • Change product repo source code when Back Office findings identify issues or product work that belongs to that repo.
  • Review generated workflow files inside the target repo before committing them. Back Office scaffolds them; it does not auto-merge them into production repos.

Recommended GitHub flow

  • Run audits from Back Office to identify work and refresh dashboards.
  • Create or update workflow definitions by scaffolding from Back Office into the target repo.
  • Open a PR in the target repo for generated workflows or code fixes.
  • Let CI validate the PR in the target repo before merge.
  • Let dashboard automation report workflow coverage back through Back Office after merge.

Scaffolded workflow files

Back Office treats GitHub Actions as a generated starting point, not a hidden control plane. The output is meant to be reviewed, edited, and committed in the product repo.

python3 scripts/backoffice-cli.py scaffold-workflows --target bible-app
  • Generated files land in the target repo under .github/workflows/.
  • The scaffold uses target-level commands from config/targets.yaml.
  • If a repo has special deploy constraints, edit the generated workflow in that repo instead of overfitting the shared template for one-off behavior.

Documentation ownership

  • The README in the Back Office repo should explain the framework, local operating model, deployment model, and links to dashboard documentation pages.
  • Product repos should keep their own product-specific setup and deployment instructions. Back Office should not replace repo-level product docs.
  • The dashboard documentation pages are the product-facing documentation surface. The README is the GitHub-facing documentation surface.
  • The metrics dashboard is the reporting surface for code-quality and agent-usage rollups: metrics.html.
